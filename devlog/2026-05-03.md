## 이슈 1 — SQL 스크립트 안에서 같은 테이블을 두 번 쓰면 이름 충돌이 남

**원인**
SQL Server는 FROM 절에 같은 테이블이 두 번 나오면 이름이 겹친다고 오류를 낸다. 많은 행을 만들기 위해 `sys.all_objects CROSS JOIN sys.all_objects` 식으로 쓰면 반드시 별명(alias)을 붙여야 한다.

**조치**
`FROM sys.all_objects a CROSS JOIN sys.all_objects b` — a, b 별명 추가

**교훈**
같은 테이블을 FROM 절에 두 번 이상 쓸 때는 항상 별명을 붙인다.

## 이슈 2 — 스크립트를 두 번 실행하면 중복 키 오류가 남

**원인**
고유 인덱스(IX_EMPG_HIS_RESULT_ID)가 걸린 테이블에 같은 RESULT_ID를 다시 INSERT 하려 해서 오류 발생. 처음 실행 후 데이터가 남아 있는 상태에서 재실행했을 때 생긴 문제.

**조치**
INSERT 앞에 `DELETE FROM ... WHERE MODEL = 'SYNTH01'` 추가 → 재실행해도 항상 같은 결과가 나오도록 만듦.

**교훈**
테스트 데이터 스크립트는 처음부터 몇 번을 실행해도 결과가 같아야 한다는 원칙으로 작성한다.

## 이슈 3 — CTE를 두 개의 INSERT에서 공유하려 했으나 두 번째가 실패함

**원인**
SQL Server의 CTE(`WITH ... AS (...)`)는 바로 뒤에 오는 단 하나의 문장에서만 유효하다. 첫 번째 INSERT가 CTE를 소비하고 나면 두 번째 INSERT는 CTE를 찾지 못해 오류가 난다.

**조치**
CTE 대신 임시 테이블(`SELECT ... INTO #base`)을 사용. 임시 테이블은 같은 연결 세션 안에서 여러 문장이 공유할 수 있다.

**교훈**
두 개 이상의 INSERT/UPDATE가 같은 중간 결과를 써야 한다면 CTE 대신 `#임시테이블`을 쓴다.

## 이슈 4 — 이력 조회 72초: nvarchar 파라미터 vs datetime2 컬럼 타입 불일치

**현상**
4월 1일 ~ 5월 3일(약 64,000건) 조회 시 앱에서 72초 소요.  
SSMS에서 동일 쿼리 실행 시 6.5초(인덱스 정상 동작 확인).

**원인 분석 과정**
1. 처음에 UPDATE_TIME 컬럼이 nvarchar라고 잘못 판단 → nvarchar 파라미터로 변경하는 오진단
2. SSMS `INFORMATION_SCHEMA.COLUMNS` 쿼리로 실제 타입 확인 → **datetime2**
3. SSMS는 literal 문자열(`'2026-04-01'`)을 쓰므로 SQL Server가 자동으로 datetime2로 추론 → 인덱스 탐
4. 앱은 `string` 파라미터(`SqlDbType.NVarChar`) → datetime2 컬럼과 타입 불일치 → **non-sargable 조건 → Full Scan → 72초**

**진단 근거**
- SSMS: `논리적 읽기 1,691` (Clustered Index Seek)
- 앱: 전체 페이지 읽기 (Full Scan) → cold buffer + 물리적 읽기 폭증

**최종 조치**

| 파일 | 수정 내용 |
|---|---|
| `QueryExecution.AddParameter()` | `DateTime` 값 → `SqlDbType.DateTime2`로 명시 (이전 버그: `SqlDbType.DateTime` 추론) |
| `Inquiry_OP200_ResourceLotHistoryViewModel.QueryEmpg()` | `startDate.ToString(...)` (nvarchar) → `startDate` (DateTime, AddParameter가 DateTime2 변환) |
| `MapRow()` | `r["UPDATE_TIME"].ToString()` → `((DateTime)r["UPDATE_TIME"]).ToString("yyyy-MM-dd HH:mm:ss.fff")` |
| `sql/07_clustered_index.sql` | UPDATE_TIME CLUSTERED INDEX 추가 (Key Lookup 제거, 범위 스캔 최적화) |

**교훈**
- `INFORMATION_SCHEMA.COLUMNS`로 실제 컬럼 타입을 먼저 확인하고 C# 파라미터 타입을 맞춘다.
- `new SqlParameter(name, (object)DateTime값)`은 `SqlDbType.DateTime`으로 추론됨. datetime2 컬럼에는 반드시 `SqlDbType.DateTime2`를 명시한다.
- SSMS literal vs 앱 파라미터의 실행계획이 다를 수 있다. 성능 차이가 크면 파라미터 타입부터 의심한다.
