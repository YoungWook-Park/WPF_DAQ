# 작업 일지 — 2026-05-07

## 작업 요약

DB 인덱스 전략 분석 및 클러스터형 인덱스 적용. 실제 프로덕션 테이블 스키마(EMPG_HIS, EMPG_HISLOG) 기준으로 인덱스 설계를 검토하고, 비클러스터형 단순 인덱스의 한계 원인을 분석함. 클러스터형 인덱스 적용 후 약 25% 성능 개선 확인(2000ms → 1509ms, 10.5만 건 기준).

---

## 이슈 1 — 비클러스터형 단순 인덱스가 성능 개선 효과가 없었던 이유

**현상**
2개월 범위 조회 시 비클러스터형 단순 인덱스 유무 차이가 거의 없음 (1.11초 vs 1.115초).

**원인**
결과셋이 전체 행의 20~30% 초과 시 SQL Server 옵티마이저가 인덱스를 무시한다 (Tipping Point).
"비클러스터형 인덱스 Seek + N만 번 RID Lookup" 비용이 Full Scan보다 비싸다고 판단하기 때문.

**교훈**
비클러스터형 인덱스는 고선택도 쿼리(전체의 5% 미만)에 효과적이다.
대량 범위 조회(날짜 기간 + 전체 컬럼 SELECT)는 클러스터형 인덱스가 정답이다.

---

## 이슈 2 — 커버링 인덱스가 이 쿼리에 불필요한 이유

**상황**
이력 조회 쿼리(`BuildSelectCols`)가 SELECT하는 컬럼: UPDATE_TIME, REPAIR, MODEL, MAT_SERIAL01, MAT_SERIAL02, TOTAL_JUDGE, APD01~44, SP01~50 — 약 100개 컬럼.

**결론**
INCLUDE에 100개 컬럼을 넣으면 테이블을 통째로 복사하는 것과 같다 → 인덱스 크기 폭증, INSERT 비용 증가.
커버링 인덱스는 SELECT 컬럼이 3~5개 수준일 때 효과적이다.
현재 구조에서는 단순 클러스터형 인덱스 + Key Lookup이 올바른 접근이다.

---

## 이슈 3 — 클러스터형 인덱스 적용 후 25% 개선에 그친 이유

**측정 결과**
- 10.5만 건 (1월 1일 ~ 4월 30일) 조회
- 인덱스 없음: 약 2000ms
- 클러스터형 인덱스: 약 1509ms → **약 25% 개선**

**원인 분석**
인덱스가 극적인 효과(수십~수백 배)를 내려면 결과셋이 작아야 한다.
10.5만 건은 전체 데이터의 상당 비율 → 인덱스는 시작 위치만 찾아줄 뿐, 읽어야 할 데이터 양은 동일.
나머지 시간은 앱 레이어 비용: `MapRow()`가 100개 필드짜리 객체를 10.5만 번 생성 (인덱스로 해결 불가).
또한 힙 테이블이었어도 시간순 INSERT로 물리 페이지가 이미 어느 정도 정렬되어 있어 Full Scan 비용이 예상보다 낮았음.

**교훈**
대량 결과셋 + 거의 모든 컬럼 조회 패턴은 인덱스보다 페이지네이션이나 조회 컬럼 축소가 더 효과적이다.

## 이슈 4 — nvarchar UPDATE_TIME이었다면 예상 성능

| 조건 | 3개월 예상 시간 |
|---|---|
| datetime + 클러스터형 (현재) | ~1.5초 |
| nvarchar + 문자열 파라미터 + 클러스터형 | ~3~5초 (키 6배 넓음, 문자 비교) |
| nvarchar + DateTime 파라미터 (타입 불일치) | ~90초 (devlog 2026-05-03 실측 기반, non-sargable Full Scan) |
